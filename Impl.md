# Leave Management system

#### **Feature: Time-Off Requests, Accruals, and Provincial ESA Compliance**

**Objective:** Enable employees to request time off (Vacation, Sick Leave, Statutory Leaves) and automate approval routing while using a backend compliance engine to enforce provincial Employment Standards Acts (ESA) minimums based on the employee's province.

#### **1\. UI / UX Components**

- **Employee Leave Dashboard:** \* A visual card layout showing current balances: "Vacation Days Available," "Sick Leave Remaining," and "Paid/Unpaid Personal Days."
  - A "Request Time Off" modal with a dropdown for leave type, date range picker, and a text area for notes.
- **Admin / Approver Calendar & Queue:**
  - A unified team calendar (drag-and-drop or grid view) displaying overlapping leaves to prevent understaffing.
  - An approval queue where managers can click "Approve" or "Deny" with automated status updates sent to the employee.

#### **2\. Backend Logic & Database Schema**

- **Accrual Engine:** A background worker running on a cron schedule (e.g., bi-weekly or monthly aligned with payroll) to calculate and add vacation hours to the employee's ledger.
- **The Provincial Policy Factory:** The system must evaluate the employee's `province_id` to determine leave logic rather than relying on a single hardcoded company policy.

#### **3\. Developer Compliance Rules (Provincial ESA Core Triggers)**

The backend must enforce minimum statutory thresholds based on the jurisdiction of the employee:

##### **Rule A: Statutory Paid Sick Leaves (BC vs. Ontario)**

- **IF Province \= British Columbia (BC):**
  - _System Rule:_ The database must hardcode a minimum of 5 days of **paid** sick leave and 3 days of **unpaid** sick leave per calendar year after 90 consecutive days of employment.
  - _Validation Logic:_ If the employer attempts to configure a global policy offering 0 paid sick days for BC staff, the frontend must block the save action and display: _"Error: BC Employment Standards mandate a minimum of 5 paid sick days after 90 days of employment."_
- **IF Province \= Ontario (ON):**
  - _System Rule:_ Mandate 3 days of **unpaid** sick leave, 3 days of **unpaid** family responsibility leave, and 2 days of **unpaid** bereavement leave per year.

##### **Rule B: Vacation Accrual Minimums**

- **Calculation Trigger:** The system monitors the employee's `hire_date`.
- **Standard Threshold (Provinces except SK):**
  - **Years 1–5:** Auto-configure vacation accrual at a minimum rate of **2 weeks (4% of gross wages)** per year.
  - **Year 5+:** On the 5th anniversary of the `hire_date`, the system must automatically adjust the database accrual rate to a minimum of **3 weeks (6% of gross wages)**.
- **Saskatchewan Exception:** \* **IF Province \= Saskatchewan (SK):** \* **Years 1–9:** Auto-set accrual to a minimum of **3 weeks (6%)**. \* **Year 10+:** Auto-set accrual to a minimum of **4 weeks (8%)**.

##### **Rule C: Job-Protected Statutory Leaves (Maternity / Parental / Compassionate Care)**

- **Action:** When an employee selects "Maternity/Parental Leave" or "Compassionate Care Leave," the system transitions their status to `Active - On Statutory Leave`.
- **Database Guardrail:** The system must lock the record so that the employee cannot be transitioned to a `Terminated` status without an explicit, multi-factor admin override and a mandatory compliance warning checkbox: _"I certify that this termination is completely unrelated to the employee's statutory leave of absence (Canadian Human Rights Code compliance)."_

# Document Management

#### **Feature: Automated Folder Hierarchy & AI-Driven Document Routing**

**Objective:** Provide a centralized, secure digital filing cabinet that automatically creates a standardized folder structure for every employee record and uses backend metadata routing to instantly categorize uploaded files (resumes, onboarding forms, certificates) into their respective folders.

---

### **1\. UI / UX Components (The Document Vault)**

- **The Employee Vault View:** A clean folder-directory interface accessible via an employee's profile. It mimics a traditional filesystem but is entirely locked down by permissions.
- **Global Dropzone Component:** A drag-and-drop file upload target present in multiple workflows. The user can drop any file format (PDF, JPEG, PNG, DOCX) without needing to navigate to a specific directory first.
- **File Metadata Sidebar:** Clicking a document opens a side panel displaying its automatically extracted properties: `Document Type`, `Associated Workflow`, `Upload Date`, and `Access Restriction Level`.

---

### **2\. Backend Logic, Directory Structure & Database Schema**

#### **Standardized Folder Hierarchy (The Core Schema)**

When a new employee record is created in the database, a background worker must automatically instantiate a relational or virtual folder tree mapped to that `employee_id`.

The system enforces a rigid, non-deletable default folder structure:

- `root/`
  - `01_Recruitment/` (Resumes, screening notes, reference checks, background checks)
  - `02_Onboarding_and_Tax/` (Offer letters, TD1 federal/provincial, direct deposit details)
  - `03_Compliance_and_Training/` (AODA certifications, WHMIS, signed handbook acknowledgments)
  - `04_Performance_and_PIPs/` (Annual appraisals, active or historical Performance Improvement Plans)
  - `05_Leaves_and_Medical/` (Doctor notes, statutory leave applications, accommodation requests)
  - `06_Offboarding/` (Resignation letters, ROE copies, signed exit/severance agreements)

#### **Automated Routing Engine (The Metadata Switchboard)**

Documents are rarely uploaded manually inside the folder view; they are uploaded via active platform workflows. The developer needs to build an event-driven router:

\[File Upload Event\] \---\> \[Identify Upload Source Context\] \---\> \[Apply Tag & Folder ID\] \---\> \[Commit to Database\]

- **Context Mapping Logic:**
  - **IF** file is uploaded via the **ATS/Recruitment Sub-Module**, update database record to `folder_id = 01_Recruitment` and `document_type = 'Resume'` or `'Background_Check'`.
  - **IF** file is uploaded via the **Onboarding Sub-Module** (e.g., employee submits completed tax documentation), update database record to `folder_id = 02_Onboarding_and_Tax` and `document_type = 'TD1_Form'`.
  - **IF** file is generated by the **Performance Sub-Module** (e.g., a signed PIP), update database record to `folder_id = 04_Performance_and_PIPs` and `document_type = 'PIP'`.

---

### **3\. Developer Compliance & Security Rules (Canadian Privacy Standards)**

Because an HR document vault holds highly sensitive Personal Information (PI), the backend must enforce strict data governance rules to comply with privacy frameworks like Quebec's Law 25 and general Canadian privacy laws.

##### **Rule A: Granular Role-Based Access Control (RBAC) & Folder Masking**

- **The Logic:** Access to folders must be evaluated at the database query level based on the current user's `role_id` and the folder’s classification level.
- **Developer Rules:**
  - `05_Leaves_and_Medical` and `04_Performance_and_PIPs` must be strictly masked.
  - Direct line managers (`role_id = 'Manager'`) **must be blocked** from reading files in `05_Leaves_and_Medical` (to protect employee medical privacy under human rights codes). Only `role_id = 'HR_Admin'` or `Super_Admin` can query this folder.
  - Employees (`role_id = 'Employee'`) can view their own files in folders `02`, `03`, and `04`, but they **cannot** see internal interview screening notes or background check files located in `01_Recruitment`.

##### **Rule B: Automated Retention & Destruction Timers (Law 25 Compliance)**

- **The Context:** Under Quebec Law 25, candidate and employee data cannot be kept indefinitely once the purpose for collection has ended.
- **The Technical Trigger:**
  - **Unsuccessful Applicants:** If a candidate's status in the ATS transitions to `Rejected`, a deletion countdown is stamped on all files mapped to their profile in `01_Recruitment`. After **24 months** (or a custom company retention setting), a cron job flags the file status as `Pending_Purge`.
  - **System Action:** Send a confirmation notification to HR: _"Notice: 24-month retention window met for rejected candidate files. Click here to securely purge these records to maintain Law 25 compliance."_

##### **Rule C: Immutable Document Audit Trail**

- **The Logic:** Any document containing compliance signatures or legal framework milestones cannot be deleted or replaced without an auditable footprint.
- **Developer Rules:** \* Turn on soft-deletes (`deleted_at` timestamp) globally across the `documents` table; files are never truly dropped from the storage bucket immediately without an admin override log.
  - Every view, download, update, or deletion action must insert a row into a `document_audit_trails` table detailing: `user_id`, `action_type`, `document_id`, `timestamp`, and `ip_address`.

To ensure the performance module protects the employer from common law liabilities, the backend must enforce the following technical constraints:

##### **Rule A: The Probationary Review Milestone Trigger**

- **The Logic:** The system continuously tracks the employee's hire_date.
- **The Trigger:** A background cron job flags employee records reaching exactly **60 days** of employment.
- **The Action:** The system automatically initializes a **90-Day Probationary Review** requisition for the manager and issues an urgent in-app notification:

- **The Context:** Drastically changing an employee's key job duties or metrics unilaterally can constitute "constructive dismissal" under Canadian common law.

- **The UI Warning:** The frontend must present a confirmation modal stating:

##### **Rule C: The "Watertight" PIP Validation Engine**

- **The Context:** To terminate an employee for cause due to poor performance under Canadian common law, an employer must prove they gave clear warnings, a reasonable timeframe to improve, and active support.
- **The Developer Rules for PIP Creation:**

  - **Rule 2 (Objective Quantification):** The PIP form must require at least one entry in the measurable_outcome and company_support_provided rich-text fields before the "Issue PIP" button transitions from disabled to active.
  - **Rule 3 (Dual Sign-off Architecture):** The PIP state cannot change to Active until a multi-factor cryptographic signature status is verified for **both** the manager_user_id and the employee_user_id.   - **Active Goals:** Progress bars for key metrics or objectives.
  - **Upcoming Milestones:** Timelines for the next 90-day review or annual appraisal.
  - **Historical Reviews:** A scrollable list of past, finalized performance documents.
- **Self-Evaluation Workflow:** A structured, step-by-step form where employees provide input on their own performance, wins, and areas for growth before the manager’s review is shared.
- **Feedback Exchange:** A simple, non-confrontational interface to request or receive feedback from peers (if enabled by HR).

---

### **2\. Backend Logic & Permissions Engine**

#### **A. The "Performance Release" Logic**

The system must prevent employees from seeing "internal-only" notes (e.g., compensation discussions, manager-to-manager calibration comments).

- **Developer Rule:** The database must use a `Review_Version` control.
  - `Admin_View` contains the full document (including manager-only calibration notes).
  - `Employee_View` contains only the `Shared_Fields` (Manager feedback, Goals, Scores).
- **Visibility Toggle:** The performance review workflow must have a `share_with_employee = TRUE` flag. The system backend will only query documents where this flag is set for the employee-facing portal.

#### **B. Dynamic Goal Tracking**

- **The Logic:** Goals should be updated in real-time by the employee.
- **Developer Rule:** The system must support bi-directional updates. When an employee marks a goal as "Complete" or updates a progress percentage, the backend must fire a `goal.update` notification to the direct manager for visibility.

---

### **3\. Developer Compliance & Legal Guardrails**

##### **Rule A: Data Integrity & Review Accuracy**

- **The Logic:** A performance review is a legal record.
- **Developer Rules:**
  - **Anti-Tamper:** Once a performance review is signed and finalized by both parties, the document is moved to `Archive` status and becomes **Read-Only**.
  - **Audit Log:** The system must record a time-stamped digital "Signature" from both parties. If a review is reopened (e.g., to fix a factual error), the system must store the original version and the corrected version, and log a reason for the change in the `performance_audit_logs`.

##### **Rule B: Privacy & Anti-Bias Protection**

- **The Logic:** Employees must be protected from biased or discriminatory commentary in their permanent HR record.
- **Developer Rules:**
  - **Keyword Scanning:** As an added guardrail, if a Manager enters text in the "Review Comments" field, the backend can run a lightweight sentiment/bias filter (optional) to flag unprofessional language _before_ the review is released to the employee.
  - **Transparency:** Under Canadian provincial law, employees are generally entitled to view their personnel files. The system must provide a clear "Export My Performance History" feature to ensure the employer is fulfilling their duty to provide access.

---

1. **Automated Nudge Engine:** Implement a reminder service that notifies employees 7 days before their self-evaluation is due.
2. **Review PDF Generation:** Create a "Generate Summary Report" function that allows the employee to download a professionally formatted PDF of their review for their own personal records.

# Document Management EE

#### **Feature: Self-Service Personal Document Vault**

**Objective:** Provide employees with a secure, read-only "Digital Filing Cabinet." This gives them access to their own employment records, payslips, and compliance documents, while ensuring they cannot view confidential HR notes, recruitment files, or sensitive company data.

---

### **1\. UI / UX Components**

- **The "My Documents" Portal:** A clean, simplified view within the Employee Profile.
- **Categorized Folder View:** Similar to the admin version, but filtered to show only what the employee is authorized to see:
  - `My_Employment_Agreement/`
  - `My_Pay_Statements/` (Syncs with Payroll module)
  - `My_Certificates_and_Training/`
  - `My_Performance_Reviews/` (Locked until released by HR)
- **Download/Preview Action:** Simple buttons to download PDFs or preview documents directly in the browser.

---

### **2\. Backend Logic & Permissions Engine**

#### **A. Data Isolation Logic (The "Read-Only" Gate)**

The backend must enforce a strict `Access_Control_List` (ACL) that prevents "Horizontal Scaling" of data.

- **Developer Rule:** When the API fetches documents for `user_id = 123`, the query MUST include a filter `WHERE owner_id = 123 AND folder_access_level = 'Employee_Visible'`.
- **The "Release" Trigger:** Performance reviews or PIP documents should be stored in the admin folder by default. The system status for these documents must be `Hidden_From_Employee`. Only when the HR Admin clicks "Publish/Release to Employee" does the backend update the status to `Visible_To_Employee`, triggering a notification.

#### **B. Document Sync & Notification**

- **Logic:** When HR uploads a new document (e.g., a signed bonus letter or a tax statement), the system should automatically "Push" this to the employee's portal.
- **Notification Trigger:** Upon a `document.upload` event where `visibility = 'public'`, the system sends an email/app notification: _"New document available in your portal: \[Document Name\]."_

---

### **3\. Developer Compliance & Privacy Rules**

##### **Rule A: Data Residency & Sovereignty (Law 25 & PIPEDA)**

- **The Logic:** Employees have a right to access their own data.
- **Developer Rules:**
  - **Right to Access:** The system must include an "Export My Data" button in the document portal. This function compiles all files mapped to the `employee_id` into a single, encrypted `.zip` archive for the employee to download.
  - **Audit Trail:** Even though the employee is viewing their own files, the system must log this access in the `document_audit_trails` table (e.g., `event: "Employee_Viewed_Document", file_id: "X", timestamp: "Y"`). This proves compliance with the employer’s duty to maintain records of who accessed sensitive data.

##### **Rule B: Document Integrity (Immutable Records)**

- **The Logic:** Documents like tax forms or signed contracts must not be alterable by the employee.
- **Developer Rules:**
  - All files in the employee vault are stored as **Read-Only** in the database.
  - If an employee needs to update a document (e.g., submitting a new void cheque), the system must utilize a "Submission Workflow" where the new file is uploaded as a `Pending_Review` status, which HR must approve before it replaces the old version in the vault.

---

### **4\. Strategic Outputs for Development: Employee Vault**

1. **Version Control:** If an employee's address or tax info changes, the system should keep the _old_ version of the document in the admin-only archive and display only the _current_ version to the employee to avoid confusion.
2. **Mobile Accessibility:** Ensure the preview component works perfectly on mobile devices, as employees often access their payslips or contracts on their phones.
3. **Legal Disclaimer:** Every document view should show a small footer: _"This document is a record for your personal files. If you believe any information is inaccurate, please contact the HR Department."_

This "General Utility" layer is what transforms an HR platform from a **static database** into a **dynamic workspace**. By integrating Agentic AI into these common administrative bottlenecks, you reduce the "HR-as-a-Helpdesk" burden and empower both managers and employees.

Here is the technical specification for these General AI Utility modules.

---

### **Module: AI-Driven Utility & Automation (The "Agentic Layer")**

#### **1\. Conversational Interface (The "HR Co-Pilot")**

- **The Intent Engine:** Instead of navigating menus, users type queries like _"When is my next payday?"_ or _"How many vacation days do I have left?"_.
- **Backend Logic:**
  - **Intent Mapping:** The Agent parses the intent (e.g., get_balance) and the entity (leave_type: "vacation").
  - **Secure API Fetch:** The Agent queries the Leave_Management API for the current user_id and returns a conversational response (e.g., _"You have 12 days of vacation remaining for 2026."_).
- **Security Guardrail:** The Agent must be restricted to the user_id context. It should _never_ be able to query another user's balance unless the requester has specific Admin or Manager permissions for that individual.

#### **2\. Document & Timesheet Generator**

- **The Workflow:** \* **Smart Templates:** Admins upload standard Word/PDF templates with "Placeholder Tags" (e.g., {{employee\_name}}, {{salary}}, {{effective\_date}}).
  - **Automated Injection:** When HR generates a letter (e.g., Employment Verification or Promotion Letter), the system fetches data from the employee record, injects it into the template, and produces a final, professional PDF.
- **Timesheet Logic:** The system aggregates time-tracking logs and auto-populates a standardized timesheet format, applying pay period business rules (e.g., rounding to the nearest 15 mins).

#### **3\. AI Meeting & Notetaking Space**

- **The Workflow:** An embedded AI scribe for performance reviews, exit interviews, or team syncs.
- **Technical Implementation:**
  - **Transcription:** Use a secure, SOC2-compliant transcription service.
  - **Summary Engine:** The LLM summarizes the conversation into: Key Decisions, Action Items, and Sentiments.
- **Compliance Rule:** Since these notes might contain sensitive HR data, the backend must save these as Confidential within the Document Management system, restricted to participants by default.

#### **4\. Automated Translation Service**

- **The Logic:** Canada is a bilingual nation (English/French). Global teams often need more.
- **Backend Strategy:**
  - **Auto-Detect:** When a document or policy is uploaded, the system detects the source language.
  - **Translation Bridge:** The system offers a "Translate to \[User's Language\]" button.
  - **Important:** The translated document must be stored as a _derived file_ of the original with a watermark stating: _"Machine-translated for convenience. The original English version is the legally binding document."_

---

To build these, you don't need to reinvent the wheel for every tool. Use a centralized **Agent Workflow Controller**.

### **Strategic Developer Rules**

##### **Rule A: Human-in-the-Loop Verification**

For any document generated by AI (e.g., a promotion letter), the system must **force an HR preview**. The "Download/Send" button remains disabled until the Admin clicks "Verified content accuracy."

##### **Rule B: PII Scrubbing**

Before sending any data (like timesheet details or salary info) to an LLM for summarization or analysis, the backend must use a **PII Scrubbing Layer** to replace sensitive info with placeholders (e.g., \[Name\] instead of "Jane Doe") if the LLM provider is third-party.

##### **Rule C: Version Control**

Never overwrite an original document. Every time an AI or template generates a new version (e.g., a revised Timesheet), it must be saved as a new version object in the Document_Management folder to maintain an audit history.
